# Atome LM — マイクロコントローラネイティブな三値言語モデルのアーキテクチャ

## 1. 動機

今日、「実際に話す」最小の言語モデルは、100 M〜1 B パラメータの範囲にあります。それらのモデルはどれも、2 ドルのマイクロコントローラが提供できる以上の RAM とメモリ帯域を必要とします。それらのモデルのアーキテクチャ上の選択——フルアテンション、密な FFN、マルチバンク MoE、検索拡張された経路——は、RAM が安価であるという前提の下でなされた選択です。Atome LM は逆の前提から出発します：RAM こそが、他のあらゆる考慮を支配する制約である、と。

その結果は、意図的に狭いアーキテクチャであり、キロバイト（メガバイトではなく）の作業 RAM を持つチップ上で動作する、固定形状の C99 推論エンジンとの互換性のために、エンドツーエンドで設計されています。

## 2. エンジンからの制約

Atome C99 エンジンの `atome_block_t` 構造体は、次のように固定されています：

```
norm        : LayerNorm
local_conv  : depthwise causal conv, ternary kernel
ssm         : diagonal SSM (per-channel a, b, c_out, FP32)
attn        : top-k causal attention, ternary Q/K/V
router      : ternary linear → softmax over 3 pathways
```

これら 3 つの経路出力それぞれのため、そして SSM の隠れ状態とアテンションの KV キャッシュのために、静的バッファが存在します。広い畳み込みのためのバッファはなく、密な FFN のためのバッファもなく、マルチバンクの重みへの備えもなく、三値カーネルに行ごとのスケールもありません。より広いアーキテクチャを学習して「後で収める」ことを試みれば、C 構造体を再生成する（本プロジェクトが依拠するビット単位で厳密な一致の契約を破る）か、推論時に静かに落とされるサポート外の経路を出荷するか、いずれかが必要になります。

したがって Atome LM はエンジンに正確に一致します：3 経路、テンソルごとのスケール、バイトトークナイザ、位置埋め込みなし、シーケンス長はコンパイル時に `ATOME_MAX_SEQ` で上限が定められます。

## 3. ブロック

```
x → LayerNorm → ┬─→ Local   (depthwise causal conv, k=5)        ─→┐
                ├─→ State   (diagonal SSM, O(L))                  ─→ Σ → +x
                └─→ Sparse  (top-k attention, O(L·k))             ─→┘
                        ↑              ↑
                        │              router weights r ∈ Δ per token
                        └──────────────┘
```

構造的に異なる 3 つの演算：

| # | 名前   | 演算                     | 役割                          |
|---|--------|-------------------------|------------------------------|
| 1 | Local  | depthwise 畳み込み k=5   | バイグラム、単語境界          |
| 2 | State  | 対角 SSM                 | 長距離のトピック持ち越し      |
| 3 | Sparse | top-k アテンション       | 共参照、正確な想起            |

ルーターは `TernaryLinear(d_model, 3)` に続いて softmax です。トークンごとに 3 通りの分布を生成します。ブロックの出力は、残差に、その分布の下での各経路出力の凸結合を加えたものです。

### 3.1 校正信号としてのルーターのエントロピー

トークンごとのルーター分布は、不確実性の信号を運びます：

```
H(r_t) = − Σ_i r_t,i · log r_t,i,    bounded in [0, log 3] for 3 pathways
```

高いエントロピーは、その位置にどの計算プリミティブが最も適切かをルーターが決められなかったことを意味します。この信号は構造的です——不確実性に特化した学習も追加のパラメータも要りません。Atome-LLM のエンジン既定規模（60 K パラメータ、単一の狭いコーパス）では、この信号は露出していますが、この規模での不確実性推定器としての校正はここでは評価していません。**本リリースには含まれない**より大きな 3 M パラメータのモデルでは、ルーターのエントロピーがドメイン外の入力を追跡し、トークンごとの損失と相関することを*予備的に*観測しました。ここでは**まだ再現可能でない観測**としてのみ報告し、裏付けとなる測定は将来のバージョンで公開する予定です。それを測定すること（例：ルーターのエントロピーとトークンごとの損失の間の期待校正誤差）は、別個の課題です。

`MCUBlock.router_entropy(x)` はトークンごとのエントロピーを nats で返します。`AtomeLM.router_entropies(ids)` は層ごと・トークンごとのエントロピーを `(B, L)` テンソルのリストとして返します。C エンジンの `atome_state_t` は、トークンごとのルーター重み配列 `router_w[ATOME_MAX_SEQ][ATOME_N_PATHWAYS]` を露出します——エントロピーはその上での和/対数です。

## 4. サイズと形状の予算

エンジンの既定 `#define`（`d_model=64`、`n_layers=4`、`d_head=16`、`vocab=256`、`kernel=5`）では：

- 埋め込み：256 × 64 = 16,384 trit
- ブロックごと：norm（256 FP32）+ conv（64 × 5 trit）+ SSM（3 × 64 FP32）+ Wq/Wk/Wv（16 × 64 + 16 × 64 + 64 × 64 trit）+ ルーター（3 × 64 trit）
- 最終 norm：128 FP32
- 逆埋め込み（unembed）：64 × 256 trit

trit あたり 2 ビットでパックすると、バイナリは構成に応じて 30〜60 KB の桁になります。典型的な既定値では快適に 100 KB を下回り、ローエンド STM32 の 1 MB フラッシュを大きく下回り、ESP32-S3 で利用可能な 8 MB より桁違いに小さくなります。

推論時の RAM 使用は、`atome_state_t` 内の静的バッファに支配されます：`x`、`normed`、3 つの経路出力スクラッチ配列、層ごとの SSM 隠れ状態配列、KV キャッシュ、ルーター重みバッファ、logits バッファ。既定値ではこれらの合計は数 KB です。

## 5. 本リリースに含まれないもの

- マルチバンク重み MoE はなし（エンジンはそれをサポートせず、ビット単位の厳密一致を破る）。
- 行ごとの三値スケールはなし（同じ理由）。
- 位置埋め込みはなし。ローカル畳み込みと SSM 隠れ状態が、エンジンのコンパイル時シーケンスウィンドウ内で位置を暗黙的に符号化します。
- 検索経路なし、エピソード記憶経路なし。いずれもオフチップ記憶か、対象ハードウェアと非互換な大きな RAM スクラッチ配列を必要とします。

これらは意図的な省略であり、欠落ではありません。RAM が拘束的制約となるハードウェア上で動作するための代償です。

## 6. 制約

- **規模。** 既定構成はおよそ 60 K パラメータ（`d_model=64`、`n_layers=4`）です。焦点を絞ったコーパスで狭く学習すれば範囲内で流暢に話し、広く学習すれば一貫性を欠きます。それはアーキテクチャではなく容量の反映です。より余裕が欲しければ `d_model` と `n_layers` を上げてください——例：`d_model=128`、`n_layers=6` はおよそ 600 K パラメータです。
- **シーケンス長。** エンジンのコンパイル時に `ATOME_MAX_SEQ` で上限が定められます（既定 32）。より長い生成には、増大する接頭辞を `atome_predict_next` に渡してトークンごとに生成してください——エンジンは各呼び出しで完全な接頭辞から SSM 隠れ状態を再導出し、Python ↔ C の一致を決定的に保ちます。
- **トークン化。** バイトレベル。UTF-8 のマルチバイト列は複数の位置を消費します。エンジン既定の `MAX_SEQ` では非ラテン文字に理想的ではありません。対象の文字が 1 文字あたりの平均バイト数が高い場合は、`ATOME_MAX_SEQ` を上げて再エクスポートすることを検討してください。
