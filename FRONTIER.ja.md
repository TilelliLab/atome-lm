# Atome LM — フロンティアの発見

> **更新 2026-05-11 — 944K でのスケールアップ A/B が要点を逆転させる。**
> 同じレシピ、同じ検証スライス、同じ公平性監査で、944K パラメータの vanilla
> GPT-FP32 ベースライン（950,608 パラメータ、Atome の 944,640 に対し +0.63 %）が
> 検証損失 0.9337 / ppl 2.54 に達し、944K で Atome 三値を損失で 11.4 %、
> 困惑度で 11.5 % 上回ります。下記の +22 % パラメータ公平 / +52 %
> フラッシュ公平の利得は **60K パラメータの MCU レジーム**でのみ、かつ
> そのレジームでのみ成立します。約 1M パラメータを超えると、3 経路ブロックの
> 帰納バイアスは容量を代替するのをやめ、容量を制約し始めます。
> 誠実な枠組みはこうです：*Atome の賭けは小モデルのレジーム——
> sub-1M パラメータ、MCU 級のデプロイ、ネットワークなし。* 944K の完全な
> 読み解きは [`HONEST_RESULTS.md`](HONEST_RESULTS.ja.md) を参照。
> マルチシードは保留。

**日付。** 2026-05-09。CPU のみ、GPU なし。
**ハードウェア。** 4 スレッドの CPU ボックス。PyTorch 2.x、FP32 参照経路。
**コーパス。** TinyStories 検証スライス、500 KB UTF-8（約 99.9 % ASCII）。
64 バイトのチャンクで学習/評価を 90/10 分割 → 7,030 の学習チャンク /
782 の留め置き（held-out）チャンク。
**オプティマイザ。** AdamW、lr 3e-4、batch 16、seq 64、3,000 ステップ。
**単一シード**（シード 0）。結果はシード間で複製されていません。

本文書は、Atome の三値 3 経路アーキテクチャと vanilla のデコーダのみ
Transformer（FP32）の間の、固定パラメータ数・固定フラッシュ予算での初の
りんご対りんご（apples-to-apples）A/B を報告します。最も近い公表済みの同類は
Andrej Karpathy の `Stories260K`——TinyStories で学習された 260 K パラメータの
FP32 の素の transformer です。Atome のフロンティア主張は「より小さいフラッシュ、
より良い品質、より少ない重みあたりビット数、*かつ* 2 ドルの
マイクロコントローラにデプロイ可能」です。本ページはそれらのうち最初の 3 つを
直接検証します。MCU デプロイは、ビット単位で厳密な Python ↔ C ↔ Cortex-M3
（QEMU）の一致により別途検証されます（`tests/test_qemu_parity.py` を参照）。

## 要約（TL;DR）

| モデル | パラメータ | ビット/重み | ディスク | bpb ↓ | 困惑度 ↓ |
|---|---:|---:|---:|---:|---:|
| **Atome 3 経路、三値** | **60,800** | **1.58** | **15.1 KB**¹ / **17.2 KB**² | **2.66** | **6.31** |
| Vanilla GPT、FP32（パラメータ公平） | 60,808 | 32 | 237.5 KB | 3.02 | 8.12 |
| Vanilla GPT、FP32（フラッシュ公平） | 5,968 | 32 | 23.3 KB | 3.71 | 13.10 |

¹ ATOME01、4 trit/バイト（現在の C エンジンはこれを読む）。
² ATOME02、5 trit/バイトの 3 進パッキング——14.4 % 小さく、
`log2(3) ≈ 1.585` ビット/trit という情報理論的下限に近い。Python の
エンコーダ + デコーダは本日出荷。C デコーダは将来の変更です。

## これが証明すること

1. **同一パラメータ数で、三値 3 経路アーキテクチャは
   素の transformer を困惑度で 22 %（6.31 対 8.12）上回り、
   同時にディスクを 16 倍少なく使う。**

   vanilla ベースラインは*過剰パラメータ化されていません*——60.8 K パラメータ
   （`d_model=44, n_layers=3, n_heads=4, d_ff=44`、目標の 8 パラメータ以内に
   収まるよう総当たり探索で選択）で一致させています。これは、あらゆる公開の
   極小 LM 論文（`Stories260K`、TinyStories 論文、小規模の BitNet）が用いるのと
   同じアーキテクチャで、些末な差を除きます。

2. **同一フラッシュ予算で、三値 3 経路アーキテクチャは
   素の transformer を困惑度で 52 %（6.31 対 13.10）上回る。**

   フラッシュ公平の vanilla ベースラインは `d_model=8, n_layers=2,
   n_heads=4, d_ff=24` です。Atome の ATOME01 バイナリ（15.1 KB）と
   ATOME02 バイナリ（17.2 KB）と同じ 20〜25 KB のディスク予算に収まります。

3. **1.58 ビットの重みは、同一アーキテクチャパラメータで FP32 に対し
   困惑度を約 22 % 費やす**——しかし FP32 版はフラッシュを 16 倍多く
   費やします。フラッシュがボトルネックとなるあらゆるデバイス（私たちが狙う
   すべての MCU）では三値が勝ちます。計算がボトルネックでフラッシュが
   無料のあらゆるデバイス（サーバー CPU）では、FP32 が品質で勝ちます。

4. **ATOME02 の 3 進パッキングは 1.6 ビット/trit に達し——1.585 ビット/trit という
   情報理論的下限の 1 % 以内**——同じ学習済み 60.8 K パラメータのモデルで、
   ディスクバイナリを 20.1 KB から 17.2 KB に削減します。C デコーダはなお保留。

## これが証明しないこと

- **単一シードのみ。** 3 つの数値はすべてシード 0 です。分散を推定するための
  マルチシードはまだ実行していません。22 % / 52 % の差は、この規模で典型的な
  シードノイズに比べて非常に大きいものの、分散は
  未測定です。
- **単一コーパス。** TinyStories は寛容な目標です——語彙が制限された
  短い物語。より広いドメインやコードのコーパスは、vanilla アテンションを
  優遇するかもしれません。私たちは測定していません。
- **単一の学習ホライズン。** 3,000 ステップは収束には
  遠く及びません。相対的な順位は、より多くの学習で入れ替わるか
  増幅するかもしれません。10 K ステップの実行が進行中で、要点が
  変わればこのページを更新します。
- **実シリコンなし。** すべての MCU の主張は QEMU
  Cortex-M3 で検証され、物理的な RP2040 / STM32 ハードウェアではありません。
  実シリコンでの tokens/sec と Joules/token はなお保留です。
- **Stories260K の直接比較はなお保留。** Karpathy の正確な
  セットアップは `Stories260K`、260 K パラメータ + 32 K トークンの SentencePiece
  語彙です。私たちのバイトトークナイザ + 60 K 構成は約 4 倍小さいです。真の
  `Stories260K` との apples-to-apples には、(a) 私たちが 260 K パラメータと
  SentencePiece トークナイザにスケールアップするか、(b) Karpathy の
  セットアップをバイトトークナイザで 60 K パラメータに再学習するか、が必要です。
  いずれも完了していません。

## 公表済みフロンティアとの比較

| システム | 最小の目標 | パラメータ | ビット/重み | 実 MCU？ | アーキテクチャは vanilla に勝つ？ |
|---|---|---:|---:|---|---|
| Microsoft BitNet b1.58 | サーバー CPU | 700 M – 3 B | 1.58 | 否 | （規模で互角） |
| Meta MobileLLM | スマートフォン | 125 M – 1 B | 4–8 | 否 | 是（同サイズの vanilla に対し） |
| Karpathy `Stories260K` | ノート PC / ブラウザ | 260 K | 32 | ファームウェアなし | 該当なし（vanilla ベースラインそのもの） |
| RP2040 上の llama.cpp（趣味） | RP2040 + SD | ~1 B（スワップ） | 4 | 是（遅い、SD 必要） | 未測定 |
| TFLite Micro / Edge Impulse | Cortex-M0+ | – | 8 | 是 | 言語タスクなし |
| **Atome LM（本研究）** | **Cortex-M0+、16 KB SRAM** | **60 K** | **1.58** | **QEMU 是、シリコン保留** | **パラメータ公平で +22 %、フラッシュ公平で +52 %** |

より小さく、よりビット効率が良く、*かつ*私たちが狙う予算でアーキテクチャ上
vanilla に勝ちます。私たちの知る限り、ルーティングアーキテクチャの勝利が
同一フラッシュ予算で vanilla ベースラインに対し直接測定された、最小の公表済み LM です。

## 再現

```bash
# from the repository root
PYTHONPATH=. python3 scripts/build_corpus.py --source tinystories \
    --max-bytes 500000 --output data/tinystories.txt

PYTHONPATH=. python3 scripts/run_ab_sweep.py \
    --train data/tinystories.txt --steps 3000 \
    --output ab_results.json
```

`ab_results.json` は上表と同じ数値を含みます
（PyTorch の行列積カーネルにおけるプラットフォーム依存の丸めを除く）。

## 未解決の問い / 次の推進

- **A1.** マルチシード（3 シード × 3 構成）で、22 % / 52 % の
  差の分散を推定する。
- **A2.** 3 つすべてを ≥ 10 K ステップ学習する。差は縮まるか、保たれるか、
  広がるか？
- **A3.** アブレーション：3 経路（ローカル畳み込み、対角
  SSM、top-k スパースアテンション）のどれがアーキテクチャの勝利の大部分を担うか？
  各々を落として測定する。
- **A4.** ATOME02 用の C デコーダを出荷する。他所のコード変更なしに、デモバイナリを
  20.1 KB から 17.2 KB に削る。
- **A5.** 実シリコン。エンジン + この 60.8 K ckpt で RP2040 を書き込む。
  tokens/sec、Joules/token を測定する。**主張を「フロンティア」から
  事実に変える要点数値。**
- **A6.** 強力な LLM 教師からの蒸留（フロンティアモデルが生成した、
  厳選された狭ドメインのテキスト 10 MB）を同じ 60 K Atome へ。
  未解決の問い：アーキテクチャの優位は蒸留の下で
  複利的に増すか？
- **A7.** Bug A の修正（Python `generate` ↔ C `atome_generate`
  の短プロンプト SSM 乖離）。ビット単位で厳密な一致の契約に触れる
  ——ユーザーの明示的な承認が必要。
