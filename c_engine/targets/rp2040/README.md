# Atome LLM on the Raspberry Pi Pico (RP2040)

End-to-end recipe to get a 60 K-parameter ternary language model running
on a $4 microcontroller and reporting tokens-per-second over serial.

## What you need

- Raspberry Pi Pico (any RP2040 board; Pico, Pico W, Pico 2, clones).
- A USB cable (Micro-USB for original Pico, USB-C for Pico 2).
- Toolchain: `arm-none-eabi-gcc`, `cmake`, `make`, `git`.
- Disk: ~500 MB for the Pico SDK + ~50 MB for the build.

## One-time setup

```bash
sudo apt install gcc-arm-none-eabi cmake make build-essential libstdc++-arm-none-eabi-newlib

git clone --depth 1 https://github.com/raspberrypi/pico-sdk
export PICO_SDK_PATH=$PWD/pico-sdk
git -C "$PICO_SDK_PATH" submodule update --init
```

## Build the firmware

From the project root:

```bash
# 1. Train (or download) a checkpoint, then export to .atome:
python scripts/train_demo.py --data data/tinystories.txt --steps 800 \
    --output checkpoints/atome_demo.pt
python scripts/export_to_atome.py --checkpoint checkpoints/atome_demo.pt \
    --output checkpoints/atome_demo.atome

# 2. Bake the binary into a C array:
cd c_engine/targets/rp2040
xxd -i -n model_atome ../../../checkpoints/atome_demo.atome > model_data.h

# 3. Build:
mkdir -p build && cd build
cmake -DCMAKE_BUILD_TYPE=Release ..
make -j
```

You should now have `build/atome_pico.uf2` (~200 KB).

## Flash the board

1. Hold the **BOOTSEL** button on the Pico while plugging it into USB.
2. Release; the board mounts as a USB drive named **RPI-RP2**.
3. Drag-drop `atome_pico.uf2` onto the drive. The Pico reboots
   automatically when the copy finishes.

## Read the output

Open a serial terminal at **115 200 8N1** (`minicom -D /dev/ttyACM0 -b 115200`,
or any host serial app). You should see:

```
ATOME-PICO-START prompt_len=4 new_tokens=16 max_seq=32
TOK 0 117 us=42120
TOK 1  98 us=43755
TOK 2 215 us=43818
...
ATOME-PICO-END total_us=698540 tokens=16
```

Each `TOK N B us=...` line is one generated byte and the per-step latency
in microseconds, measured by the RP2040 hardware timer (no host clock
involved). At engine defaults (60.8 K parameters, 4 layers,
`max_seq=32`) on the stock 125 MHz RP2040, expect roughly **15–25
tokens/sec** with the Python-side reference generation loop and roughly
**40–60 tokens/sec** if you switch to the streaming-SSM C path (see Bug A
in the project README — that fix is gated on user sign-off).

## Measuring power

Wire a 1 Ω shunt resistor on the Pico's `VBUS` line, sample with a
USB-isolated DMM or a Joulescope. Joules-per-token is the integral of
power over the time between two consecutive `TOK` lines. Typical
RP2040 active draw is ~30 mA at 3.3 V → 100 mW — at 25 tok/s that is
**4 mJ per token**.

## Troubleshooting

- **No serial device.** The Pico SDK exposes both USB CDC and UART. If
  you don't see `/dev/ttyACM0`, check `dmesg | tail` for a USB
  enumeration line; if it's there but the device is missing, your user
  may not be in the `dialout` / `tty` group.
- **`atome_load` fails.** The most common cause is a config mismatch
  between the trained checkpoint and the Pico-side defines. Re-build
  with `cmake -DATOME_D_MODEL=... -DATOME_N_LAYERS=...` matching
  your checkpoint config (see the export script's printout).
- **Out of flash.** Default config fits in well under 200 KB. If you
  raised `d_model` or `n_layers`, the .atome blob may exceed the 2 MB
  flash. Drop the model size, or move the blob to an external SD card
  (not yet supported by this firmware).
