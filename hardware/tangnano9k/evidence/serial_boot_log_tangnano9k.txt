*** time -- see the README section above for the corrected story and
*** what real inference output looks like after this fix.

"0A 14 05 11 00 19" is the hardcoded prompt {10,20,5,17,0,25} from

The 8 generated tokens are all 0xFF (255): expected. With random,

this placeholder's random seed, not a bug. Re-running with a real
trained checkpoint (via MODEL_ATOME=path/to/real.atome) is the next
step for a quality check against the Python reference.

NOTE ON rv32im vs rv32imc (see c_engine/targets/picorv32-tangnano9k/README.md
for the full writeup):

The original plan called for `rv32imc` (compressed ISA) for code
density. Bisected on this exact hardware: rv32imc firmware boots and
prints fine below roughly 8 KB, but silently produces zero UART bytes
once progmem grows past that -- reproduced with a firmware carrying
NO atome/libm/libgcc code at all (just a padded array), so it is a
hardware/toolchain interaction, not an atome-lm bug. rv32im does not
show this at any size tested, including this ~17 KB firmware. This
target now defaults to rv32im.

HONEST SCOPE
- Proof-of-execution artifact: the Atome engine (this repo's own
  c_engine/upstream/atome.c) booting and running to completion on
  physical PicoRV32/Tang Nano 9K silicon, over real UART.
- NOT a quality or performance benchmark -- the weights are untrained
  placeholders (see Model, above) and no cycles/token timing was
  captured in this run.
- RAM headroom (1932 B between .bss and the stack top) was not
  independently verified via a stack-painting build; the firmware
  running to completion without visible corruption is consistent with
  but not proof of that margin holding under all inputs.
