# Atome on real silicon — PicoRV32 / Tang Nano 9K

The `nano`/`tiny` config of the Atome ternary engine, booting and
running to completion on a **physical Tang Nano 9K FPGA** running a
**PicoRV32 soft-core**, over real UART, fully offline. Same engine —
[`c_engine/upstream/atome.c`](../../c_engine/upstream/) — as every
other target in this repo, unmodified.

> **Honest scope.** This is a *proof-of-execution* artifact: the
> engine loads a model, runs `atome_predict_next()` in a loop, and
> prints tokens back over UART, on real silicon. It is **not** a
> quality or performance benchmark. No trained `nano`/`tiny` checkpoint
> exists in this repo yet (see below), so the captured run uses
> structurally-valid but untrained placeholder weights — the generated
> tokens are noise by design, not a bug. No cycles/token timing was
> captured either.

Captured output (`evidence/serial_boot_log_tangnano9k.txt`):

```
atome-picorv32 boot
prompt: 0A 14 05 11 00 19
generated: FF FF FF FF FF FF FF FF
done
```

RAM headroom was also measured directly on this board, not inferred
(`evidence/serial_ramcheck_log_tangnano9k.txt`, stack-painting technique):

```
bss=0x00003874 stack_used=0x00000130 total=0x000039A4 ram_budget=0x00004000
```

14452 B bss + 304 B measured stack = 14756 / 16384 B (90.1%) — **1628 B
(9.9%) free**, for the same 6-token-prompt + 8-generation-step run shown
above. Tight, but real and positive.

## A real bug this run found: `rv32imc` silently breaks on this hardware

The original plan for this target
([`PICORV32_TANGNANO9K_PLAN.md`](../../PICORV32_TANGNANO9K_PLAN.md))
called for `rv32imc` (RISC-V compressed ISA) for code density. Bisected
on real hardware: **firmware built with `rv32imc` boots and prints
fine below roughly 8 KB of progmem, then silently produces zero UART
output once it grows past that — reproduced with a firmware carrying
no `atome.c`/`libm`/`libgcc` code at all**, so this is a
hardware/toolchain interaction (likely PicoRV32's `COMPRESSED_ISA`
fetch path vs. this `progmem.v`'s fixed single-cycle word-only read
port), not an engine bug. `rv32im` (no compressed ISA) doesn't show
this at any size tested. The
[`c_engine/targets/picorv32-tangnano9k`](../../c_engine/targets/picorv32-tangnano9k/)
Makefile now defaults to `rv32im`; see that target's README for the
full bisection writeup.

This is the kind of thing that only shows up with real hardware in the
loop — synthesis and `make check-budget` both pass silently either
way; only the actual UART output reveals it.

## A second bug this run found — and this one is upstream, not ours

After fixing a NULL-pointer aliasing bug (RAM starts at address `0x0`
on this platform; see the target README), real inference started
running for the first time and hung. Bisected with temporary UART
instrumentation (`evidence/serial_alignment_bug_log_tangnano9k.txt`):
`atome_load()`'s binary parser produces misaligned `float*` pointers
(`gamma_addr=0x001042DB`, 3 bytes off a 4-byte boundary — traced to the
7-byte `"ATOME01"` magic string never being corrected by any later
section). PicoRV32 traps misaligned loads by default and, with no
bus-error IRQ wired up on this minimal top, hangs permanently instead
of crashing or continuing. This is a real bug in
`c_engine/upstream/atome.c`'s parser, not in this board's glue code —
confirmed by the exact same checkpoint running to completion natively
on x86 (which tolerates unaligned loads in hardware) with output that
matches the Python reference bit-for-bit. See the target README's "A
third bug" section for the full writeup and the native-host proof.

where noted (nothing here needed adaptation; `top_uart_hello.v` and
the PicoRV32/UART IP are used as-is).

## Reproduce it yourself

Needs: `riscv-none-elf-gcc` (or another bare-metal RV32 GCC), Gowin
EDA (with `gw_sh` on `PATH` or its full path), `openFPGALoader`, and a
connected Tang Nano 9K.

```bash
# 1. Build the firmware (from c_engine/targets/picorv32-tangnano9k/)
cd c_engine/targets/picorv32-tangnano9k
make firmware.bin                    # MARCH=rv32im by default; uses the placeholder model
#   for a real model instead:
#   make MODEL_ATOME=path/to/real.atome firmware.bin

# 2. Bake it into progmem.v
cd ../../../hardware/tangnano9k/firmware
python3 progmem.py ../../../c_engine/targets/picorv32-tangnano9k/firmware.bin hw/src/progmem.v

# 3. Synthesize (headless Gowin -- Fedora-specific env, see FreeRTOS-TetriSaraj's
#    0.doc/milestone_0_1_uart_hello.md for why each var is needed)
cd hw
export GOWIN=/path/to/gowin/IDE/bin/gw_sh
export GOWIN_ENV='XDG_CACHE_HOME=/tmp LD_PRELOAD=/usr/lib64/libfreetype.so.6 LD_LIBRARY_PATH=/path/to/gowin/IDE/lib QT_QPA_PLATFORM=minimal QT_PLUGIN_PATH=/path/to/gowin/IDE/plugins/qt QT_QPA_PLATFORM_PLUGIN_PATH=/path/to/gowin/IDE/plugins/qt/platforms QT_XCB_GL_INTEGRATION=none LIBGL_ALWAYS_SOFTWARE=1 QT_OPENGL=software QT_QUICK_BACKEND=software'
env -i HOME="$HOME" PATH="/usr/local/bin:/usr/bin:/bin" $GOWIN_ENV $GOWIN build.tcl

# 4. Flash (SRAM -- volatile, safe to redo; drop `-f` from openFPGALoader for
#    persistent flash, not needed for iteration)
openFPGALoader -b tangnano9k impl/pnr/uart_hello.fs

# 5. Read the UART (channel B of the FT2232 -- confirmed /dev/ttyUSB1 on this
#    board via `openFPGALoader --scan-usb`; ttyUSB0 is JTAG)
python3 -m serial.tools.miniterm /dev/ttyUSB1 115200
```

If step 5 shows nothing: check you're on `rv32im` (see the bug above),
confirm `/dev/ttyUSB1` vs `ttyUSB0` for your board with
`openFPGALoader --scan-usb`, and check `dialout` group membership for
serial port permissions.
