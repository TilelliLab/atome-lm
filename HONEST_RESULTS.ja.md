¹ ATOME01、4 trit/バイトのパッキング。  
² テンソルごとの部分は 2.81 ビット/重み。行ごとの α は出力行ごとに FP16 を 1 つ
追加します（944K では % のオーバーヘッドは無視できる）。

**加工なしの要点：**

- 60K の MCU 目標では、三値 3 経路アーキテクチャは**同一パラメータ数で困惑度 22 %**、**同一フラッシュ予算で 52 %** vanilla FP32 を上回ります。
- 944K では、素の三値は**検証損失で 11.4 %、困惑度で 11.5 % vanilla FP32 に負けます**。同じレシピ、同じ検証スライス、同じシード。
- 944K の逆転は、このキットで最も重要な誠実な発見です。それが示すのは：3 経路の帰納バイアスは小規模では容量を代替し、より大きな規模では容量を制約するということです。Atome の賭けは小モデル / MCU のレジームであり——「極小の三値が何にでも勝つ」ではありません。

## 表 4 — 何が堅固で、何が「支えているが薄い」か

**堅固（強い理由なく変えないこと）：**

- HEAD で 146/146 のテストが緑（うち 16 は power3 専用）。
- 単一 forward の Python ↔ C ↔ Cortex-M3（QEMU）のビット単位で厳密な一致
  （`tests/test_parity_with_c.py`、`tests/c_parity/parity_main.c`）。
- 学習済みの atome_1m_v1.pt + vanilla_1m_v1.pt の成果物がディスク上にあり、いずれも
  `checkpoints/*.train.json` に完全な学習ログを持つ（開いてください——各ステップの
  損失が記録されています）。
- 60K のパラメータ公平 / フラッシュ公平 A/B は約 30 分の CPU で再現可能
  （`scripts/run_ab_sweep.py`）。

**支えているが薄い：**

- すべての要点数値は単一シード。
- マルチトークンの C 生成には以前、SSM 状態の発散バグ
  （Bug A）がありました。Python と C エンジンの双方で修正済み：`atome_predict_next`
  は各呼び出しで SSM 隠れ状態をリセットし、完全なトークン接頭辞から再導出します
  （`c_engine/upstream/atome.c`）。マルチトークンの Python↔C 一致は
  `tests/test_parity_multitoken.py` でカバーされ、単一 forward の一致は
  `tests/test_parity_with_c.py` によりビット単位で厳密なままです。
- RP2040 のデモは現在、944K で 264 KB SRAM を超えます——MCU の主張は
  レジーム依存であり、このキットのランチャーは
  power3 がパラメータ予算を十分に狭めて 944K を射程に戻せるかを検証中です
  （単独では無理で、Q15 かより小さい隠れ状態が必要）。

## 記録ファイル

学習済みの 944K チェックポイントとその学習ログはキットとともに出荷されるため、
報告される各数値はステップ単位で監査可能*かつ*直接再評価可能です：

- `checkpoints/atome_944k.bin` — パック済みの C エンジン blob（ATOME01 形式）。
- `checkpoints/atome_1m_v1.pt` — 944K Atome の PyTorch ソース。
- `checkpoints/vanilla_1m_v1.pt` — 944K vanilla FP32 ベースライン（上記の
  逆転 A/B 用）。
- `checkpoints/atome_1m_v1.train.json` — 1000 ステップごとの学習ログ。
- `checkpoints/vanilla_1m_v1.train.json` — vanilla ベースラインについて同様。
- `ab_results.json` — 正確な 60K A/B の結果行。
- `FRONTIER.md` — 944K の完全開示を含むフロンティアの解説。
- `PAPER.md` — アーキテクチャの解説。
- `tests/` — 146 個の緑のテスト。

60K スイープ自体（`checkpoints/ab_sweep/`）は**出荷されません**——それは
24 回の使い捨ての学習実行でした。スイープの再現には、同梱の `scripts/` を用いて
CPU で約 20 分かかります。
